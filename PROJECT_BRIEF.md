# مرشدي الوظيفي (CareerPilot) — بريف مشروع معسكر هندسة وكلاء الذكاء الاصطناعي

## المشكلة والمستخدم المستهدف
- **المشكلة**: البحث عن وظيفة عملية مبعثرة ومرهقة — تبحث في مواقع متعددة،
  تقيّم يدوياً هل الوظيفة تناسب سيرتك الذاتية، تكتب خطاب تقديم من الصفر
  لكل وظيفة، وتفقد تتبع أي وظيفة قدّمت عليها ومتى تحتاج تتابع.
- **المستخدم المستهدف**: باحث عن عمل (طالب متخرج أو موظف يبحث عن فرصة جديدة)
  يقدّم على عدة وظائف أسبوعياً ويحتاج نظاماً يسرّع ويرتّب هذي العملية.

## الحل
وكيل ذكاء اصطناعي واحد يتحدث معه المستخدم بلغة طبيعية (عربي/إنجليزي)،
يقرر بنفسه أي أداة يحتاج من 7 أدوات متاحة له: اقتراح مسميات وظيفية مناسبة
من تحليل السيرة الذاتية، بحث عن وظائف، تقييم توافق الوظيفة مع السيرة
الذاتية، اقتراح تعديل السيرة الذاتية (خاصة الـ Summary) لتناسب وظيفة
معينة، صياغة خطاب تقديم، تسجيل التقديمات، ومتابعتها. لا يُرسل أو يُسجّل
أو يُعدّل أي ملف نهائي إلا بموافقة صريحة من المستخدم.

## سير العمل (يطابق نموذج المعسكر الرسمي)
```
طلب المستخدم (رفع PDF للسيرة الذاتية + نص حر، عربي أو إنجليزي)
   ↓
تحليل الطلب (استخراج نص PDF، Gemini يفهم النية ويحلل السيرة)
   ↓
اختيار الأداة المناسبة (الوكيل يقرر من بين 7 أدوات، وقد يستدعي أكثر من واحدة بالتسلسل)
   ↓
تنفيذ المهمة (مثال: suggest_job_titles → المستخدم يختار/يكتب → search_jobs → evaluate_match)
   ↓
عرض النتيجة (نص مباشر، وينتظر موافقة المستخدم قبل أي حفظ نهائي)
```

### ملاحظة عن مصدر بيانات الوظائف (Jooble)
Jooble محرك تجميع (aggregator) يسحب إعلانات من مصادر متعددة (ATS، LinkedIn،
وغيرها) — نفس فكرة Google Jobs. موثوق كمُجمّع، وله تغطية فعلية للسعودية،
لكن أضف رسالة تنبيه بسيطة في مخرجات الوكيل: "تحقق من شرعية الشركة قبل
التقديم الفعلي" — هذا يعكس وعياً بمصداقية البيانات وتفصيل جيد للعرض.

## دور الـ AI Agent بالتفصيل
1. **فهم النية** من نص حر غير مقيّد بأوامر ثابتة
2. **اقتراح مسميات وظيفية**: يحلل السيرة الذاتية (بعد استخراجها من PDF)
   ويقترح 3-5 مسميات وظيفية مناسبة، ويسمح للمستخدم يختار من القائمة أو
   يكتب مسمى آخر بنفسه — الاقتراح لا يُفرض، هو بداية حوار
3. **اتخاذ قرار تحليلي حقيقي**: تقييم مدى توافق وظيفة مع سيرة ذاتية ليس بحثاً
   نصياً بسيطاً، بل تحليل دلالي يتطلب فهم سياق (الخبرة، المهارات، التخصص)،
   وبناءً عليه اقتراح صياغة أفضل لقسم الـ Summary تحديداً لتلك الوظيفة
4. **تنفيذ متعدد الخطوات**: طلب واحد قد يستدعي أكثر من أداة بالتسلسل
   (اقترح مسميات → ابحث → قيّم → اكتب خطاب) دون أن يطلب المستخدم كل خطوة
5. **Human-in-the-loop**: أي إجراء نهائي (حفظ، اعتماد خطاب) ينتظر موافقة صريحة

## الأدوات المستخدمة
| الأداة | الغرض | نوعها |
|---|---|---|
| Gemini API (`gemini-2.5-flash`) + Function Calling | العقل المدبّر للوكيل: فهم النية، اختيار الأداة، التحليل، الكتابة، اقتراح تعديل السيرة | نموذج ذكاء اصطناعي |
| Jooble API | جلب إعلانات وظائف حقيقية (مجاني، بدون تعقيد OAuth) | API خارجي |
| تخزين محلي (JSON أو SQLite) | سجل التقديمات (الشركة، المسمى، الحالة، التاريخ) | قاعدة بيانات بسيطة |
| Python (venv) | بيئة التشغيل | أداة تطوير |
| `pypdf` أو `pdfplumber` | استخراج نص السيرة الذاتية من ملف PDF يرفعه المستخدم | مكتبة بايثون |
| (اختياري لاحقاً) TTS من مشروع سابق | قراءة صوتية لملخص التقديمات | إعادة استخدام كود جاهز عندك |

## محتوى الشرائح المقترح
1. اسم المشروع + الشعار
2. المشكلة (البحث عن وظيفة مرهق ومبعثر)
3. المستخدم المستهدف
4. فكرة الحل (وكيل واحد يقرر ويتصرف نيابة عنك، بموافقتك)
5. سير العمل (مخطط الأسهم الخماسي)
6. الأدوات (Gemini Function Calling، Jooble API، تخزين محلي)
7. لقطات فعلية من حوار حقيقي مع الوكيل
8. التحديات (مثال: دقة تقييم التوافق، تنسيق قرارات الوكيل متعددة الخطوات)
9. التطوير المستقبلي (ربط بـ LinkedIn، تحليل فجوة المهارات، تذكيرات متابعة تلقائية)
10. شكراً

## بيانات حساسة — ملاحظة
سيرتك الذاتية بيانات شخصية. لا تعرضها كاملة في الشرائح أو الفيديو — استخدم
سيرة ذاتية تجريبية (Demo Resume) ببيانات وهمية واقعية عند التسجيل والعرض.

## معايير القبول
- الوكيل يفهم طلبات حرة غير مقيّدة بصيغة أوامر ثابتة
- سلسلة استدعاء أدوات متعددة تعمل بدون تدخل يدوي بين كل خطوة
- لا حفظ نهائي بدون تأكيد صريح من المستخدم
- تعمل الأدوات بشكل مستقل (فشل واحدة لا يوقف البقية)
